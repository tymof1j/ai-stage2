Відтворюваний звіт

Дані, рішення, формули, backtesting та обмеження

Цей звіт документує pipeline, який створює три основні сторінки Air Alert Atlas. Усі числові артефакти генеруються командою python3 scripts/build_data.py, а сайт — quarto render.

python3 -m pip install -r requirements.txt
make check
make data
quarto render

1. Pitch: чому це корисно для deftech / miltech

Проєкт розглядає повітряні тривоги як відкритий operational signal: не для прогнозування ударів, а для вимірювання навантаження системи попередження, оцінки політики локалізації та побудови відтворюваного OSINT-пайплайну.

Практична цінність для defense tech:

  • Operational awareness: короткий прогноз 3–6 годин показує очікуване агреговане навантаження на систему оповіщення, що можна використовувати для планування моніторингу, комунікацій і уваги команди.
  • Policy analytics: контрфакт локалізації вимірює, скільки район-хвилин тривоги не було б оголошено завдяки більш точному рівню попередження.
  • Auditability: усі числа походять з публічного CSV, тестованої interval algebra, часових split-ів і задокументованих обмежень, а не з “чорної скриньки”.

Для цивільних користь м’якша, але важлива: архів робить видимим історичний burden тривог, нічне навантаження та різницю між регіонами. Водночас сайт навмисно не дає tactical або safety-рекомендацій і не замінює офіційні сповіщення.

2. Джерела та snapshot

Інтервали взято з публічного official_data_en.csv, реконструйованого з офіційного каналу повітряних тривог. Локальна копія має checksum, дату завантаження і coverage у data/raw/README.md.

Адміністративні межі — OCHA Ukraine Common Operational Dataset v05, valid from 1 September 2025. Для браузера геометрію спрощено сервером ArcGIS; назви прив’язано через adm1_pcode та adm2_pcode.

Відомі прогалини

  • Офіційний набір починається 15 березня 2022 року.
  • Постійні тривоги Луганщини та Криму не представлені звичайними інтервалами.
  • Hromada-level записи для районної аналітики агреговано до батьківського району.
  • Чотири невалідні/незавершені рядки snapshot відкинуто.

3. Інтервальна алгебра

Усі інтервали мають семантику [start, end). Перед сумуванням вони сортуються й об’єднуються, тому паралельні записи області, району та громади не подвоюють тривалість.

Час переводиться з UTC до Europe/Kyiv; інтервали, що перетинають місцеву північ, розділяються. Денна тривалість — union minutes, а не сума сирих рядків.

Основні метрики:

  • кількість початків об’єднаних епізодів;
  • унікальні хвилини під тривогою;
  • найдовший денний сегмент;
  • хвилини в інтервалі 22:00–06:00.

Регресійні тести для overlap, adjacency, midnight split і hourly allocation розташовані у tests/test_intervals.py.

4. Контрфактична локалізація

Дата початку аналізу — 1 грудня 2025 року. Кожен alert проектується на канонічне районне покриття:

  • обласний запис охоплює всі райони області;
  • районний — один район;
  • громадський — батьківський район.

Для області з (N_o) районами та хвилини (t):

\[ S_{o,t} = N_o \cdot I(A_{o,t} > 0) - A_{o,t}, \]

де (A_{o,t}) — кількість районів із фактичною тривогою. Підсумок (S) — район-хвилини, яких не було б у моделі суцільного обласного попередження.

Це не оцінка “врятованого часу” чи правильності рішення ППО. Результат однаково зважує райони, не населення, і описує лише різницю між двома політиками оповіщення на наявних інтервалах.

5. Snapshot forecast

Target

Погодинне навантаження:

\[ L_t = \frac{\text{район-хвилини під тривогою в годині }t} {60 \times \text{кількість моніторених районів}}. \]

Луганщину, Крим і Севастополь виключено з denominator через нерепрезентовані постійні режими.

Features

  • load lags 0–24, 48, 72, 168 годин;
  • rolling mean 3/6/12/24/72/168 годин;
  • rolling standard deviation 6/24/168 годин;
  • частота початків і відбоїв за 6/24 години;
  • циклічні hour-of-day та day-of-week.

Для кожного горизонту 1–6 годин навчається окрема standardized ridge regression. alpha обирається на 30-денному validation window; фінальна оцінка — наступні 30 днів.

Головний categorical outlook для наступних 3–6 годин — середнє прогнозоване навантаження на горизонтах +3, +4, +5 і +6 годин. Позначки “низьке / помірне / середньо-високе / високе” відповідають квартилям погодинного навантаження у training-періоді до validation window; отже це відносна історична категорія, а не оцінка небезпеки.

Baselines:

  1. поточне навантаження зберігається;
  2. навантаження тієї самої майбутньої години вчора.

Інтервал прогнозу побудовано з 90-го перцентиля абсолютних validation residuals. Це емпіричний model-error interval, не повна оцінка геополітичної невизначеності.

Live API limitation

Репозиторій не містить API token. Отримання персонального доступу потребує заявки й схвалення. Тому прогноз прив’язаний до останньої повної години локального CSV snapshot і стає застарілим одразу після появи нових подій.

Production ingestion мав би:

  1. отримувати активні статуси після кожної зміни;
  2. оновлювати поточну неповну годину;
  3. переобчислювати features;
  4. публікувати новий snapshot із timestamp;
  5. кешувати відповіді й дотримуватися rate limit.

Прогноз не можна використовувати замість офіційної системи оповіщення.

6. Використання AI

AI був основним engineering collaborator:

  • пошук і перевірка bulk data замість непридатного history API;
  • проєктування interval algebra та контрфакту;
  • створення тестів і перевірка leakage;
  • вибір temporal validation замість випадкового split;
  • frontend architecture та accessibility review;
  • документування припущень і limitations.

Кожне AI-рішення, що впливає на числа, закріплено кодом, тестом, формулою або backtest-метрикою. Сам LLM не генерує прогнозів.
